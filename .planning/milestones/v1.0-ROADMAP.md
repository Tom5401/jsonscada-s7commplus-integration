# Milestone v1.0: S7CommPlus Alarm Subscriptions PoC

**Status:** ✅ SHIPPED 2026-03-18
**Phases:** 1
**Total Plans:** 2

## Overview

Extended the existing S7CommPlusClient driver to receive native S7CommPlus alarm subscription events and persist them to MongoDB. Fixed 5 protocol-level bugs blocking alarm subscriptions, built a dedicated alarm thread with full lifecycle management, and validated end-to-end with a live S7-1500 PLC + PLCSIM Advanced. All 17 v1 requirements delivered in a single cohesive phase.

## Phases

### Phase 1: End-to-End Alarm Pipeline

**Goal**: A running driver that receives S7CommPlus alarm notifications from a live PLC, processes Coming and Going events without crashing or dropping events silently, and inserts structured documents into a `s7plusAlarmEvents` MongoDB collection — while tag polling continues unaffected

**Depends on**: Nothing (first phase)
**Plans**: 2 plans

## Milestone Summary

**Key Accomplishments:**
- Fixed BUG-01/02/05: credit-limit renewal targets correct ObjectId, subscription slot freed cleanly on disconnect, RelationId collision with tag subscription eliminated
- Fixed BUG-03/04: unknown PDU types caught without crashing alarm thread; ack-only notifications skipped gracefully without null dereference
- AlarmThread.cs built as dedicated partial class with full lifecycle: connect → subscribe → receive loop → MongoDB write → unsubscribe → disconnect
- End-to-end validated live: Coming and Going events confirmed in s7plusAlarmEvents; tag polling unaffected
- Rich alarm document schema: alarmText, infoText, additionalTexts (SD_1–SD_9 with @N%x@ placeholder resolution), associatedValues (typed SD_1–SD_10), priority, alarmClass, groupId, allStatesInfo, timestamps

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Separate MongoDB collection `s7plusAlarmEvents` | Alarm events are time-series log entries, not live point state | ✓ Good — clean separation confirmed |
| Extend S7CommPlusClient (not new project) | Cohesive PoC; alarm subscription belongs alongside tag driver | ✓ Good — single build, shared config |
| Read-only ack | Reduces PoC scope; write-back can be added later | — Pending for future milestone |
| Credit-limit replenishment inside WaitForAlarmNotification | Keeps alarm thread body minimal; protocol knowledge stays in driver layer | ✓ Good — alarm thread is simple loop |
| Separate S7CommPlusConnection for alarm thread | Avoids shared state and PDU interleaving with tag connection | ✓ Good — no threading issues observed |
| InsertOneAsync + GetAwaiter().GetResult() | Sync-over-async acceptable for low-frequency alarm events in sync thread | ✓ Good — no performance issues |
| LCID 1033 (English) hardcoded | Matches project language; multi-language is out of scope for PoC | — Pending for future milestone |

**Issues Resolved:**
- Alarm thread exiting immediately on first 5-second idle timeout (WaitForNewS7plusReceived ERROR logged as fatal; fixed by clearing m_LastError on timeout)
- Spurious "ERROR: Timeout!" console noise every 5 seconds when no alarm present
- Additional texts showing raw @1%f@ placeholder instead of resolved float value (SD values were parsed but never substituted)

**Issues Deferred:**
- Alarm acknowledgement write-back to PLC (out of scope for PoC)
- Multi-language alarm text support (LCID hardcoded to 1033)
- Auto-resubscribe after connection loss (reconnect handling out of scope for v1)
- Unit tests for alarm code (requires live PLC; accepted as PoC trade-off)

**Technical Debt Incurred:**
- No formal VERIFICATION.md — verified via live PLC UAT + integration checker
- Post-execution enhancements (timeout fix, additionalTexts, placeholder resolution) not captured in SUMMARY frontmatter
- Join(3000) shorter than 5s WaitForAlarmNotification timeout — alarm thread may outlive bounded join (non-defect, still cleans up)
