# Requirements Archive: v1.1 — Alarm Management & Viewer

**Archived:** 2026-03-23
**Core Value:** Alarms from S7-1200/S7-1500 PLCs appear in json-scada via native protocol subscription — not polling — with full metadata (text, timestamp, ack state, associated values), so operators see real events the moment they occur.

---

## v1.1 Requirements (All Validated)

### Driver & Data

- [x] **DRVR-01**: MongoDB `ackState` field correctly reflects PLC acknowledgement state — `false` for unacknowledged, `true` for acknowledged
  - *Outcome:* Fixed DateTime.MinValue → DateTime.UnixEpoch sentinel in AlarmThread.cs. Confirmed via PLCSIM trace. Validated in Phase 2.
- [x] **DRVR-02**: MongoDB alarm documents include a resolved `alarmClassName` string derived from the numeric alarm class ID
  - *Outcome:* AlarmClassNames dictionary added to AlarmThread.cs with AlarmClass 33 = "Acknowledgment required" and Unknown (N) fallback. Validated in Phase 2.
- [x] **DRVR-03**: Acknowledgement command sent from json-scada is received and applied by the PLC
  - *Outcome:* Full ack pipeline delivered in Phase 4. SendAlarmAck() via alarmConn confirmed against PLCSIM Advanced V8. TIA Portal shows alarm acknowledged.

### Alarm Viewer

- [x] **VIEW-01**: User can navigate to a dedicated S7Plus Alarms Viewer page in AdminUI, separate from the existing tag-based alarm viewer
  - *Outcome:* /s7plus-alarms route registered, Dashboard tile added. Existing /alarms-viewer untouched. Validated in Phase 3.
- [x] **VIEW-02**: Alarm viewer displays TIA Portal-equivalent columns — Source, Date, Time, Status, Acknowledge, Alarm class name, Event text, ID, Additional texts 1–3
  - *Outcome:* 11-column Vuetify table with custom slot templates. Validated in Phase 3.
- [x] **VIEW-03**: User can acknowledge an unacknowledged alarm from the viewer
  - *Outcome:* Ack button with spinner, POST /ackS7PlusAlarm endpoint, C# ack pipeline. Full round-trip confirmed via UAT (2026-03-23, 4/4 tests passed). Validated in Phase 4.
- [x] **VIEW-04**: Alarm viewer automatically refreshes to show new alarms without manual page reload
  - *Outcome:* setInterval(fetchAlarms, 5000) in onMounted, clearInterval in onUnmounted. Validated in Phase 3.
- [x] **VIEW-05**: User can filter displayed alarms by status (Incoming / Outgoing / All)
  - *Outcome:* statusFilter ref drives filteredAlarms computed. Validated in Phase 3.
- [x] **VIEW-06**: User can filter displayed alarms by alarm class
  - *Outcome:* alarmClassFilter ref with dynamic alarmClassOptions computed from live data. Validated in Phase 3.
